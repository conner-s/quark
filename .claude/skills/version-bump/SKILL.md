# version-bump

Quark's version string lives in **six** places. The project guidance (CLAUDE.md, README) only names three — the README shields.io badge, the `Cargo.lock` `quark` entry, and the iOS `Info.plist` are easy to forget, and they have historically been bumped late in separate commits (the plist drifted to `0.11.3` while the tree was at `0.12.0`). This skill changes all six together and refuses to run if they're already out of sync.

| # | File | Where the version appears |
|---|------|---------------------------|
| 1 | `package.json` | `"version": "X.Y.Z"` |
| 2 | `src-tauri/Cargo.toml` | `[package]` → `version = "X.Y.Z"` (first occurrence only — **not** dependency versions) |
| 3 | `src-tauri/tauri.conf.json` | `"version": "X.Y.Z"` |
| 4 | `README.md` | shields.io badge: `version-X.Y.Z-<color>` |
| 5 | `src-tauri/Cargo.lock` | the `quark` package block (also regenerated by any `cargo build`, but synced here so the tree is clean immediately) |
| 6 | `src-tauri/gen/apple/quark_iOS/Info.plist` | `CFBundleShortVersionString` **and** `CFBundleVersion` (both `<string>X.Y.Z</string>`) |

## When to bump (timing)

**Bump once per branch, when prepping it for merge — not eagerly while the feature is still in progress.** A single branch often lands several feature/fix commits; bumping on each one churns the six version files, produces noisy intermediate versions, and is the main source of the cross-file drift this skill exists to prevent. So:

- ✅ The branch is otherwise ready to merge (work complete, tests green) → bump once, with the **highest** SemVer level any commit on the branch warrants (one feature commit anywhere ⇒ minor, even if the rest are fixes).
- ❌ Mid-feature, just after a single commit, or "to be safe" before more work lands on the same branch → don't bump yet.
- If you're unsure whether the branch is at merge-prep, ask the user before bumping.

This timing supersedes any "bump whenever shipping a feature or fix" phrasing — the per-commit reading caused the drift, so we bump per-merge instead. (CLAUDE.md's Versioning section is aligned with this.)

## SemVer rule (from CLAUDE.md)

- **Feature** — new user-visible behaviour → bump **minor** (`0.4.1 → 0.5.0`)
- **Bug fix** — corrects existing behaviour, no new features → bump **patch** (`0.4.1 → 0.4.2`)
- **major** is reserved for breaking/milestone releases — only when the user explicitly asks.

If you're unsure whether the change is a feature or a fix, ask the user before bumping.

## How to run

From the repo root:

```bash
bash .claude/skills/version-bump/bump.sh <major|minor|patch|X.Y.Z>
```

Examples:

```bash
bash .claude/skills/version-bump/bump.sh minor    # shipped a feature
bash .claude/skills/version-bump/bump.sh patch     # shipped a bug fix
bash .claude/skills/version-bump/bump.sh 1.0.0      # explicit version
```

The script:
1. Reads the current version from `package.json` (the source of truth).
2. **Aborts if the five files don't already agree** — printing which one drifted, so you reconcile an existing mismatch rather than burying it.
3. Computes the new version (or takes the explicit one).
4. Rewrites all five with scoped substitutions (dependency versions in Cargo.toml/Cargo.lock are left alone).
5. Verifies each file now reports the new version and prints the changed lines.

It only edits the working tree — it does **not** commit, tag, or push. Review `git diff`, then commit as usual.

## Notes & gotchas

- **Don't hand-edit just one file.** That's the exact drift this skill exists to prevent. Always go through the script.
- **Cargo.lock**: the script updates only the `quark` entry. A later `cargo build` will reproduce the same change, so either order is fine.
- **`reconcile to 'X' before bumping`**: the script found the files disagreeing. Set them all to the intended current version first (run the script with that explicit `X.Y.Z`, or fix by hand), then bump.
- The script needs `perl` and `awk` (both available in the project's nix dev shell and on the host).
- It does not touch `DESIGN.md` changelog entries or git tags — handle those separately if your release flow needs them.
